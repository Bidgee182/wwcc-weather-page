Setup Guide

WWCC Daily Report
GitHub Actions Setup

Step-by-step instructions for uploading files to GitHub and configuring automated daily email reports.

1

What You Are Uploading

The complete folder and file structure for the GitHub Actions system

The WWCC_GitHub_Actions folder on your Desktop contains everything needed. Upload the entire contents (not the folder itself) to the root of your wwcc-weather-page repository.

wwcc-weather-page/ <-- your existing GitHub repo ├── .github/ │ └── workflows/ │ └── daily-report.yml <-- the automation schedule ├── data/ │ ├── daily_log.csv <-- permanent data archive (starts empty) │ └── reports/ │ └── .gitkeep <-- keeps the folder tracked by git ├── scripts/ │ ├── daily_report.py <-- the main reporting script │ └── requirements.txt <-- Python packages (requests, sendgrid) └── weather.html <-- your existing dashboard (already there)
Note: The data/daily_log.csv file already contains the header row. Each morning the script will append one new row of data. Never delete this file or you will lose your historical record.
2

Upload Files to GitHub

Using the GitHub website (no command line required)

The easiest way to upload is directly through the GitHub website:

  1. Go to github.com and open your wwcc-weather-page repository.
  2. Click Add file → Upload files.
  3. Open the WWCC_GitHub_Actions folder on your Desktop.
  4. Drag and drop all the files and folders (.github, data, scripts) into the upload area on GitHub.
  5. Scroll down, add a commit message such as "Add GitHub Actions daily report system", and click Commit changes.
Important: The .github folder starts with a dot. On Windows, this folder may be hidden. If you cannot see it, open File Explorer, go to the View tab, and tick Hidden items. Then drag it to GitHub normally.
Alternative — GitHub Desktop: If you have GitHub Desktop installed, copy the files into your local clone of the repo, then commit and push from the app. This is often easier for folders with dots in the name.

Add GitHub Secrets

Store your API credentials and email settings securely in GitHub

GitHub Secrets keep your passwords and API keys out of your code. The daily report script reads these automatically each time it runs.

To add a secret:

  1. Go to your repository on GitHub.
  2. Click Settings (top right of the repo, not your account).
  3. In the left sidebar, click Secrets and variables → Actions.
  4. Click New repository secret.
  5. Enter the Name exactly as shown below (capitals matter), paste in the Value, and click Add secret.
  6. Repeat for every row in the table.
Secret Name Value to Enter What It Is
DAVIS_DID 001D0A00AB84 Davis station device ID (v1 API)
DAVIS_PASS your Davis WeatherLink password Davis account password (v1 API)
DAVIS_TOKEN your Davis v1 API token Davis v1 API token
DAVIS_V2_KEY your Davis v2 API key Davis WeatherLink v2 API key
DAVIS_V2_SECRET your Davis v2 API secret Davis WeatherLink v2 API secret (for HMAC signing)
DAVIS_V2_STATION 10489 Davis station ID on the v2 API
SENDGRID_API_KEY your SendGrid API key SendGrid key for sending emails (starts with SG.)
EMAIL_FROM weather@wwcc.com.au The address emails are sent from (must be a verified sender in SendGrid)
EMAIL_GK_RECIPIENTS greenkeeper@wwcc.com.au Comma-separated list of greenkeeper email addresses. Receives the daily morning briefing, weekly, and monthly summaries.
EMAIL_COMMITTEE_RECIPIENTS secretary@wwcc.com.au,captain@wwcc.com.au Comma-separated list of committee/management addresses. Receives weekly and monthly summaries only (not the daily briefing).
SendGrid sender verification: Before emails can be sent, the EMAIL_FROM address must be verified in SendGrid as an approved sender. Log in to sendgrid.com, go to Settings → Sender Authentication, and verify the address. Without this step, all emails will be rejected.
4

Test the Workflow

Trigger a manual run to confirm everything is working before the first scheduled run

  1. Go to your repository on GitHub.
  2. Click the Actions tab at the top.
  3. In the left sidebar, click WWCC Daily Weather Report.
  4. Click Run workflow → Run workflow (green button).
  5. A new run will appear. Click on it to watch the live log output.
  6. If all steps show a green tick, the email has been sent and the CSV has been updated.
  7. Check your inbox for the morning briefing email.
If a step fails: Click on the failing step to expand its log. The most common issues are a missing secret (check spelling exactly) or SendGrid sender not verified. The script prints clear error messages to help diagnose the problem.

Automatic Schedule

What gets sent, to whom, and when

Daily — Every Morning
6:00 AM AEST
Morning briefing to greenkeeper only
Weekly — Every Monday
6:00 AM AEST
7-day summary to all recipients
Monthly — 1st of Each Month
6:00 AM AEST
Full month summary to all recipients
Data Archive
Every Day
One row added to daily_log.csv — permanent record

The workflow uses two cron entries to handle Australian daylight saving automatically:

  • AEST (May to October): runs at 19:10 UTC = 6:10 AM AEST
  • AEDT (October to April): runs at 18:10 UTC = 5:10 AM AEDT

Both entries run year-round; one will be off-season but harmless (the duplicate-send guard in the script prevents double sending). The 10-minute offset compensates for typical GitHub Actions queue delays.

No manual DST adjustment needed: the dual-cron setup handles AEST/AEDT switching automatically. If you need to change the delivery time, update both cron entries in .github/workflows/daily-report.yml.

What Each Email Contains

Summary of data included in the daily, weekly, and monthly reports

Daily Morning Briefing (greenkeeper only)

  • Yesterday's temperature range, mean humidity, rainfall, and evapotranspiration
  • GDD accumulation for Bentgrass (base 10°C) and Kikuyu (base 15°C), plus 7-day running totals
  • Leaf wetness hours, Delta T mean, UV index peak
  • Disease risk levels — Dollar Spot (Smith-Kerns %), Fusarium, Brown Patch, Pythium
  • Soil moisture balance (7-day rain minus ET) and zone classification
  • Spray condition breakdown — hours classified as GO / CAUTION / NO-GO
  • Disease alert flag if any high-risk condition was triggered

Weekly Summary (all recipients — sent every Monday)

  • 7-day table showing all key metrics for each day
  • Weekly GDD totals for both grass types
  • Disease risk trend across the week

Monthly Summary (all recipients — sent 1st of each month)

  • Full month of data in a condensed table
  • Cumulative GDD and rainfall for the month
  • Disease alert count for the month

Accessing the Data Archive

How to view, download, and use the permanent CSV record

Every day the script appends one row to data/daily_log.csv in your GitHub repository. This file grows continuously and becomes your permanent weather and disease record.

To download the full archive:

  1. Go to your repository on GitHub.
  2. Click the data folder, then click daily_log.csv.
  3. Click the download button (arrow icon, top right of the file view).
  4. Open the downloaded file in Microsoft Excel or Google Sheets.

To filter or analyse the data in Excel:

  • Select the header row and click Data → Filter to filter by date range, disease risk level, or soil zone.
  • Use Insert → Chart to plot GDD accumulation, rainfall, or disease risk over time.
  • The disease_alert column contains YES or NO — filter on YES to see all high-risk days.
Individual daily HTML reports are also saved to data/reports/YYYY/MM/YYYY-MM-DD.html in the repository. These are the same formatted HTML email that was sent that morning — useful for looking back at a specific day in detail.

SendGrid Free Account Setup

Creating a free email delivery account if you do not have one yet

  1. Go to sendgrid.com and sign up for a free account.
  2. The free tier allows 100 emails per day — more than enough for this system.
  3. Once logged in, go to Settings → API Keys.
  4. Click Create API Key, give it a name like WWCC Weather Bot, select Restricted Access, and enable Mail Send permissions only.
  5. Copy the key (it starts with SG.) and save it as the SENDGRID_API_KEY GitHub Secret.
  6. Go to Settings → Sender Authentication and verify the email address you want to send from (the value you set as EMAIL_FROM). SendGrid will send a verification email to that address — click the link to confirm.
You must verify the sender address. If the from-address is not verified in SendGrid, every email will be silently rejected and you will see a 403 error in the GitHub Actions log.
